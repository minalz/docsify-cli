# 🤖 AI Agent 架构详解

> 🧠 本文整理 ReAct、ReWOO、SDD、SPEC 四种 AI Agent 相关架构与方法论的核心概念与对比。

---

## 🔄 ReAct（Reason + Act，推理+行动）

### 💡 核心思想

让大语言模型（LLM）**交替进行"思考"和"行动"**，形成推理与工具调用的闭环。

### 📝 工作流程

```
循环：
  Thought（思考） -> 分析当前状态，决定下一步
  Action（行动）  -> 调用工具（搜索、计算、数据库等）
  Observation（观察） -> 获取工具返回结果
  Thought（思考） -> 基于观察继续推理
  ...直到任务完成
```

### 💻 代码示例

```python
from langchain.agents import create_react_agent

agent = create_react_agent(llm, tools, prompt)
result = agent.invoke({"input": "北京今天天气如何？适合穿什么衣服？"})
```

### 🎯 适用场景

- 需要多步推理的复杂查询
- 工具调用之间存在依赖关系
- 中间结果会影响后续决策

### ⚖️ 优点与缺点

| 优点 | 缺点 |
|------|------|
| 推理过程透明，可解释性强 | 需要多次 LLM 调用，延迟较高 |
| 能动态调整策略 | 工具调用串行执行，效率受限 |
| 容错性好，观察错误可修正 | 长任务可能陷入循环 |

## 📋 ReWOO（Reasoning WithOut Observation，无观察推理）

### 💡 核心思想

**预先生成完整的执行计划**，一次性规划所有工具调用，无需中间观察反馈。

### 🆚 与 ReAct 的对比

| 维度 | ReAct | ReWOO |
|------|-------|-------|
| 执行方式 | 逐步执行，观察后决策 | 一次性生成完整计划 |
| 工具调用 | 串行，多轮交互 | 可并行执行 |
| LLM 调用次数 | 多（每步一次） | 少（规划+求解两次） |
| 适用场景 | 动态、不确定环境 | 计划明确、可预测任务 |
| 延迟 | 较高 | 较低 |

### 📝 工作流程

```
Planner（规划器）
  -> 生成完整计划（包含所有工具调用模板）
Worker（执行器）
  -> 并行执行所有工具调用
Solver（求解器）
  -> 整合结果，生成最终答案
```

### 💻 示例

```
规划器输出：
  步骤1：调用 weather_api(city="北京") -> weather_result
  步骤2：调用 dress_advisor(weather=weather_result) -> advice
  步骤3：整合 weather_result 和 advice 生成最终回答

执行器并行执行步骤1和2
求解器填充模板生成答案
```

### 🎯 适用场景

- 批量数据处理
- 工具调用相互独立
- 对延迟敏感的应用

## 🗂️ SDD（Structured Dialogue Definition，结构化对话定义）

### 💡 核心思想

用**结构化定义**规范多轮对话流程，实现精确的对话状态管理。

### 🧩 核心组件

```yaml
dialogue:
  nodes:
    - id: ask_destination
      prompt: "请问您要去哪里？"
      slots:
        - name: destination
          type: string
          required: true
      transitions:
        - condition: "slot_filled(destination)"
          next: ask_date

    - id: ask_date
      prompt: "请问出发日期是？"
      slots:
        - name: date
          type: date
          required: true
      transitions:
        - condition: "slot_filled(date)"
          next: confirm_booking

    - id: confirm_booking
      prompt: "确认预订 {destination}，日期 {date}？"
      action: "execute_booking"
```

### 🎯 应用场景

- 智能客服机器人
- 表单自动填写助手
- 医疗问诊对话系统
- 银行/保险业务办理

### 🆚 与 ReAct/ReWOO 的区别

| | ReAct/ReWOO | SDD |
|--|-------------|-----|
| 灵活性 | 高，动态决策 | 低，预定义流程 |
| 可控性 | 较低 | 高，严格流程约束 |
| 适用 | 开放域任务 | 封闭域、合规场景 |

## 📐 SPEC（多义缩写，需结合上下文）

> 注：SPEC 在不同上下文中有不同含义，以下为 AI Agent 领域常见解释。

### 📝 解释一：结构化规划与执行链（Structured Planning and Execution Chain）

将任务分解为**规范化的规划-执行单元**：
- **S**tructured：结构化输入输出
- **P**lanning：显式规划阶段
- **E**xecution：独立执行阶段
- **C**hain：链式组合

### 📝 解释二：系统提示工程规范（System Prompt Engineering and Control）

通过精心设计的系统提示（System Prompt）约束 LLM 行为：

```
系统提示示例：
  你是一名专业的数据分析助手。请遵循以下规范：
  1. 所有计算必须使用工具完成，禁止臆造数据
  2. 输出必须包含数据来源引用
  3. 不确定时回答"无法确认"，禁止猜测
```

### 📝 解释三：软件规格说明（Software Specification）

传统软件工程术语，指需求文档、接口定义、技术规格等。

## 📊 综合对比

| 架构 | 核心范式 | 规划方式 | 执行方式 | 最佳场景 |
|------|---------|---------|---------|---------|
| **ReAct** | 推理+行动交替 | 动态规划 | 串行 | 探索性任务、复杂推理 |
| **ReWOO** | 规划与执行分离 | 静态预规划 | 可并行 | 批量任务、低延迟需求 |
| **SDD** | 结构化状态机 | 预定义流程 | 按流程执行 | 合规对话、业务流程 |
| **SPEC** | 规范化约束 | 视具体定义 | 视具体定义 | 可控性要求高的场景 |

---

## 💡 实践建议

1. **任务简单、需要透明推理** -> 用 **ReAct**
2. **任务复杂、工具调用多、追求效率** -> 用 **ReWOO**
3. **业务流程固定、需要合规控制** -> 用 **SDD**
4. **需要严格约束 LLM 行为** -> 结合 **SPEC** 规范
