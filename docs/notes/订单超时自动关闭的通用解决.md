## 📋 一、问题背景

在电商、支付、预约等系统中，用户下单后需要在规定时间内完成支付，否则订单应自动关闭。常见场景：

| 场景 | 超时时间 | 业务动作 |
|------|---------|---------|
| 电商订单 | 30分钟 | 未支付自动取消，库存回滚 |
| 秒杀订单 | 15分钟 | 未支付释放库存，防止超卖 |
| 预约服务 | 24小时 | 未支付释放时段，他人可约 |
| 优惠券领取 | 72小时 | 过期自动失效 |

**核心挑战**：
1. **精准触发**：倒计时为0时立即执行，不能早也不能晚
2. **高可靠**：不能漏单、不能重复执行
3. **高并发**：大促期间海量订单同时到期
4. **资金安全**：已支付的订单不能被错误关闭

## 📱 三、Layer 1: 前端倒计时（用户体验层）

### 3.1 设计目标

- **提升用户体验**：用户看到实时倒计时，感知系统响应
- **减少无效请求**：倒计时结束前不频繁轮询后端
- **纯前端实现**：不涉及业务逻辑，仅UI展示

### 3.2 实现方案

```javascript
// 下单成功后，后端返回过期时间戳
const expireTime = response.data.expireTime; // 1721649000000

// 前端JS倒计时
function startCountdown(expireTime) {
    const timer = setInterval(() => {
        const now = Date.now();
        const remaining = expireTime - now;

        if (remaining <= 0) {
            clearInterval(timer);
            // UI变化：按钮变灰，显示"已关闭"
            document.getElementById('payBtn').disabled = true;
            document.getElementById('payBtn').textContent = '已关闭';
            document.getElementById('status').textContent = '订单已关闭';
            // 注意：不调用取消API，仅UI变化
            return;
        }

        // 格式化显示：29:59
        const minutes = Math.floor(remaining / 60000);
        const seconds = Math.floor((remaining % 60000) / 1000);
        document.getElementById('countdown').textContent = 
            `${String(minutes).padStart(2, '0')}:${String(seconds).padStart(2, '0')}`;
    }, 1000);
}
```

### 3.3 关键要点

| 要点 | 说明 |
|------|------|
| **不调用API** | 倒计时到0仅UI变化，不真正调用取消接口 |
| **时间同步** | 前端时间可能不准，以服务器返回的过期时间为准 |
| **刷新处理** | 页面刷新后重新获取剩余时间，继续倒计时 |

## 🔴 四、Layer 2: Redis ZSet 时间轮（精准触发层）

### 4.1 为什么用 Redis ZSet？

| 方案 | 精度 | 可靠性 | 复杂度 | 适用场景 |
|------|------|--------|--------|----------|
| RocketMQ 延时消息 | 分钟级 | 高 | 低 | 固定延迟级别 |
| **Redis ZSet 时间轮** | **秒级/毫秒级** | **高** | **中** | **通用推荐** |
| 数据库定时扫描 | 秒级 | 高 | 中 | 兜底方案 |
| 内存时间轮 | 毫秒级 | 低（易丢） | 高 | 短延时 |

### 4.2 核心原理

```
Redis ZSet 天然有序集合：
  ZADD order:timewheel 1721649000000 "order:12345"
         ↑ key           ↑ score(执行时间戳)   ↑ member(任务ID)

查询到期任务：
  ZRANGEBYSCORE order:timewheel 0 1721649000000 LIMIT 0 100
  → 只返回 score <= 当前时间的任务

原子移除（分布式竞争）：
  ZREM order:timewheel "order:12345"
  → 返回1表示我抢到了，返回0表示被其他实例抢走了
```

### 4.3 完整代码实现

```java
@Component
@Slf4j
public class RedisTimeWheel {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String TIME_WHEEL_KEY = "timewheel:tasks";
    private static final String LOCK_PREFIX = "timewheel:lock:";
    private static final long SCAN_INTERVAL_MS = 100; // 100ms扫描一次

    /**
     * 添加延时任务
     * @param taskId 任务唯一标识
     * @param delayMs 延迟毫秒数
     * @param taskData 任务数据（JSON）
     */
    public void addTask(String taskId, long delayMs, String taskData) {
        long executeTime = System.currentTimeMillis() + delayMs;

        String taskInfo = JSON.toJSONString(Map.of(
            "taskId", taskId,
            "executeTime", executeTime,
            "data", taskData,
            "createTime", System.currentTimeMillis()
        ));

        // ZADD：score = 执行时间戳
        redisTemplate.opsForZSet().add(TIME_WHEEL_KEY, taskInfo, executeTime);

        log.info("添加延时任务: taskId={}, 执行时间={}, 延迟={}ms", 
            taskId, executeTime, delayMs);
    }

    /**
     * 核心扫描逻辑：每100ms执行一次
     */
    @Scheduled(fixedRate = SCAN_INTERVAL_MS)
    public void scanAndExecute() {
        long now = System.currentTimeMillis();

        // 查询已到期的任务（score <= now），每次最多取100条
        Set<ZSetOperations.TypedTuple<String>> tasks = redisTemplate.opsForZSet()
            .rangeByScoreWithScores(TIME_WHEEL_KEY, 0, now, 0, 100);

        if (tasks == null || tasks.isEmpty()) {
            return; // 没有到期任务，直接返回（无开销）
        }

        for (ZSetOperations.TypedTuple<String> taskTuple : tasks) {
            String taskInfo = taskTuple.getValue();

            // 原子移除：利用 ZREM 的返回值判断谁抢到了
            Long removed = redisTemplate.opsForZSet()
                .remove(TIME_WHEEL_KEY, taskInfo);

            if (removed == null || removed == 0) {
                continue; // 被其他实例抢走了
            }

            // 异步执行（不阻塞扫描线程）
            TaskInfo task = JSON.parseObject(taskInfo, TaskInfo.class);
            executorService.submit(() -> executeTask(task));
        }
    }

    /**
     * 执行任务
     */
    private void executeTask(TaskInfo task) {
        String lockKey = LOCK_PREFIX + task.getTaskId();

        // 双重保险：分布式锁防止重复执行
        if (!redisLock.tryLock(lockKey, 0, 30)) {
            log.warn("任务已被其他实例执行: {}", task.getTaskId());
            return;
        }

        try {
            log.info("执行任务: taskId={}, 计划时间={}, 实际时间={}, 误差={}ms",
                task.getTaskId(),
                task.getExecuteTime(),
                System.currentTimeMillis(),
                System.currentTimeMillis() - task.getExecuteTime()
            );

            // 执行业务逻辑
            if (task.getTaskId().startsWith("order:")) {
                orderService.cancelIfUnpaid(task.getTaskId().replace("order:", ""));
            }

        } catch (Exception e) {
            log.error("任务执行失败: {}", task.getTaskId(), e);
            // 转入死信队列（见下方）
        } finally {
            redisLock.unlock(lockKey);
        }
    }

    @Data
    static class TaskInfo {
        private String taskId;
        private long executeTime;
        private String data;
        private long createTime;
    }
}
```

### 4.4 取消任务（用户主动支付时）

```java
/**
 * 维护 taskId -> member 的映射，O(1) 取消
 */
public void addTask(String taskId, long delayMs, String taskData) {
    long executeTime = System.currentTimeMillis() + delayMs;
    String taskInfo = JSON.toJSONString(...);

    // 事务：同时写入 ZSet 和 Hash 映射
    redisTemplate.execute(new SessionCallback<Object>() {
        @Override
        public Object execute(RedisOperations operations) throws DataAccessException {
            operations.multi();
            operations.opsForZSet().add(TIME_WHEEL_KEY, taskInfo, executeTime);
            operations.opsForHash().put("timewheel:map", taskId, taskInfo);
            return operations.exec();
        }
    });
}

public boolean cancelTask(String taskId) {
    String taskInfo = (String) redisTemplate.opsForHash().get("timewheel:map", taskId);
    if (taskInfo == null) return false;

    // 事务：同时移除
    redisTemplate.execute(new SessionCallback<Object>() {
        @Override
        public Object execute(RedisOperations operations) throws DataAccessException {
            operations.multi();
            operations.opsForZSet().remove(TIME_WHEEL_KEY, taskInfo);
            operations.opsForHash().delete("timewheel:map", taskId);
            return operations.exec();
        }
    });
    return true;
}
```

### 4.5 性能优化

| 优化点 | 方案 | 效果 |
|--------|------|------|
| 批量处理 | 每次扫描 LIMIT 100 | 减少Redis往返次数 |
| 异步执行 | 线程池异步处理 | 不阻塞扫描线程 |
| 原子竞争 | ZREM 返回值判断 | 天然分布式锁 |
| 空转优化 | 无任务立即返回 | 降低CPU消耗 |

## 🗄️ 五、Layer 3: 数据库兜底扫描（容错层）

### 5.1 为什么需要兜底？

Redis 可能故障、任务可能丢失、ZREM 可能失败。数据库是最终可信源。

### 5.2 实现方案

```java
@Component
@Slf4j
public class OrderFallbackJob {

    @Autowired
    private OrderMapper orderMapper;

    /**
     * 每30秒扫描一次，处理漏网之鱼
     */
    @Scheduled(fixedRate = 30000)
    public void fallbackScan() {
        // 查询30分钟前创建且未支付的订单
        List<Order> expiredOrders = orderMapper.selectExpiredUnpaidOrders(
            LocalDateTime.now().minusMinutes(30), 100);

        for (Order order : expiredOrders) {
            // 分布式锁（防止多实例并发）
            String lockKey = "order:cancel:" + order.getId();
            if (!redisLock.tryLock(lockKey, 0, 10)) continue;

            try {
                // 双重校验：可能时间轮已经处理过了
                if (order.getStatus() == OrderStatus.UNPAID) {
                    cancelOrder(order.getId());
                    log.warn("兜底扫描发现过期订单: {}", order.getId());
                }
            } finally {
                redisLock.unlock(lockKey);
            }
        }
    }
}
```

### 5.3 SQL 实现

```sql
-- 查询过期未支付订单（带行级锁，防止并发）
SELECT * FROM orders 
WHERE status = 'UNPAID' 
  AND create_time < DATE_SUB(NOW(), INTERVAL 30 MINUTE)
ORDER BY create_time ASC
LIMIT 100
FOR UPDATE SKIP LOCKED;

-- MySQL 8.0+ 支持 SKIP LOCKED
-- PostgreSQL 也支持 SKIP LOCKED
-- 作用：跳过已被其他事务锁定的行，避免阻塞
```

### 5.4 索引优化

```sql
-- 联合索引：create_time 在前，status 在后
CREATE INDEX idx_orders_create_time_status 
ON orders(create_time, status);

-- 或覆盖索引（避免回表）
CREATE INDEX idx_orders_create_status_id 
ON orders(create_time, status, id);
```

## 📊 六、Layer 4: T+1 对账补偿（最终一致性）

### 6.1 为什么需要对账？

即使有三层保障，极端情况下仍可能出现：
- 用户已支付，但订单被错误关闭（支付回调延迟）
- 分布式事务不一致（支付成功但订单状态未更新）

### 6.2 实现方案

```java
@Component
@Slf4j
public class DailyReconciliationJob {

    @Autowired
    private OrderMapper orderMapper;
    @Autowired
    private PaymentService paymentService;
    @Autowired
    private AlertService alertService;
    @Autowired
    private CompensationService compensationService;

    /**
     * 每日凌晨2:30执行对账
     */
    @Scheduled(cron = "0 30 2 * * ?")
    public void reconcile() {
        // 查询昨日"已关闭"状态的订单
        List<Order> closedOrders = orderMapper.selectClosedOrdersYesterday();

        for (Order order : closedOrders) {
            // 去支付系统核对：是否真的没付款？
            PaymentStatus paymentStatus = paymentService.query(order.getId());

            if (paymentStatus == PaymentStatus.PAID) {
                // 漏单！用户付了钱但订单被关了
                log.error("对账发现漏单: orderId={}, 需补偿", order.getId());
                alertService.sendAlert("漏单告警", order.getId());

                // 自动补偿：重新开启订单或退款
                compensationService.compensate(order);
            }
        }
    }
}
```

### 6.3 补偿策略

| 场景 | 补偿动作 |
|------|---------|
| 已支付但订单关闭 | 重新开启订单，通知用户 |
| 已发货但订单关闭 | 拦截物流，退款 |
| 无法重新开启 | 原路退款，短信通知用户 |

## ❓ 八、常见问题

### Q1: 100ms扫描频率太高，会不会压垮Redis？

**不会。** ZRANGEBYSCORE 是有序集合的范围查询，时间复杂度 O(logN + M)，N是总数据量，M是返回结果数。如果没有到期任务，M=0，几乎无开销。

### Q2: 如果任务执行耗时超过100ms，会不会堆积？

**不会。** 扫描和执行是分离的：
- 扫描线程：只负责查到期任务，提交到线程池，立即返回
- 执行线程：线程池异步处理，不阻塞扫描

### Q3: 多实例部署时，如何避免重复执行？

**ZREM原子竞争：**
```
实例A: ZREM key member → 返回1（抢到，执行）
实例B: ZREM key member → 返回0（没抢到，跳过）
```
天然分布式锁，无需额外锁组件。

### Q4: Redis挂了怎么办？

**Layer 3兜底：** 数据库每30秒扫描，即使Redis完全不可用，订单也能在30秒内被关闭。同时触发Redis故障告警，运维介入恢复。

### Q5: 用户支付的同时订单被关闭，怎么办？

**幂等设计：**
```java
public void cancelOrder(String orderId) {
    // CAS更新：只有 UNPAID 才能变为 CLOSED
    int updated = orderMapper.updateStatus(orderId, "UNPAID", "CLOSED");
    if (updated == 0) {
        // 可能已支付，不处理
        log.warn("订单状态已变更，取消失败: {}", orderId);
        return;
    }
    // 回滚库存...
}
```
