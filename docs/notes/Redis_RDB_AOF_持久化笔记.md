## 📋 一、核心概念速览

| 维度 | **RDB（Redis Database）** | **AOF（Append Only File）** |
|:---|:---|:---|
| **本质** | 内存快照（Snapshot） | 命令日志（Command Log） |
| **文件格式** | 紧凑的二进制 `.rdb` | 文本格式 `.aof`（可配置混合模式） |
| **触发方式** | 手动（SAVE/BGSAVE）或条件触发 | 实时追加，三种刷盘策略可选 |
| **数据安全** | 可能丢失最后一次快照后的数据 | 更高，最多丢失 1 秒（everysec） |
| **恢复速度** | **极快**，直接加载二进制 | 较慢，需逐条重放命令 |
| **文件体积** | 紧凑，经压缩后很小 | 较大，但支持重写压缩 |
| **运行时开销** | BGSAVE 时 fork 子进程，有短暂阻塞 | 持续写盘开销，重写时 fork |

## 💾 二、RDB 深入解析

### 2.1 工作原理

RDB 通过 `fork()` 系统调用创建子进程，由子进程将当前内存数据写入磁盘生成 `.rdb` 文件，主进程继续处理客户端请求。

```
┌─────────────┐     fork()      ┌─────────────┐
│  Redis主进程 │ ─────────────→ │   子进程     │
│  (继续服务)  │                │ 写RDB到磁盘  │
└─────────────┘                └─────────────┘
```

### 2.2 Copy-On-Write（写时复制）

- `fork()` 后父子进程共享内存页表
- 只有当内存页被修改时，才会真正复制该页
- **风险**：写入频繁时，COW 导致内存暴增，可能触发 OOM

### 2.3 触发条件

```conf
# redis.conf 自动触发示例
save 900 1      # 900秒内至少1次修改
save 300 10     # 300秒内至少10次修改
save 60 10000   # 60秒内至少10000次修改
```

### 2.4 适用场景

- 数据备份与灾难恢复（冷备份）
- 快速重启恢复大内存实例
- 对数据完整性要求不高的缓存场景

## 📝 三、AOF 深入解析

### 3.1 工作原理

每条写命令执行后，先追加到 AOF 缓冲区，再根据配置策略刷入磁盘。

```
客户端写命令 → 追加到 AOF 缓冲区 → 根据策略刷盘
                                    │
              ┌─────────────────────┼─────────────────────┐
              ▼                     ▼                     ▼
        always (每条)           everysec (每秒)         no (OS决定)
        最安全，最慢              推荐，平衡               最快，最不安全
```

### 3.2 三种刷盘策略对比

| 策略 | 说明 | 数据安全 | 性能影响 |
|:---|:---|:---|:---|
| `always` | 每条命令都 fsync | 零丢失 | 吞吐量下降 50%+ |
| `everysec` | 每秒后台线程 fsync | 最多丢 1 秒 | 推荐，性能与安全平衡 |
| `no` | 由操作系统决定 | 可能丢几十秒 | 最高吞吐，不推荐生产 |

### 3.3 AOF 重写（Rewrite）

AOF 文件会随时间膨胀，需要定期重写压缩：

- **重写原理**：基于当前内存数据生成最小命令集，而非读取旧 AOF
- **触发方式**：手动 `BGREWRITEAOF` 或自动条件触发
- **配置示例**：

```conf
auto-aof-rewrite-percentage 100  # AOF 增长 100% 时触发
auto-aof-rewrite-min-size 64mb   # 最小重写阈值
```

### 3.4 适用场景

- 数据安全性要求高的业务（金融、订单）
- 需要精确恢复到某个时间点的场景
- 作为 RDB 的补充，提供增量持久化

## 🔀 四、Redis 7.0+ 混合持久化（重要更新）

### 4.1 文件结构

```
AOF 文件 = RDB 前缀（二进制）+ AOF 增量部分（文本命令）

┌─────────────────┬─────────────────────┐
│   RDB 前缀       │     AOF 增量部分     │
│ (二进制，快速加载) │ (文本命令，保证安全)  │
└─────────────────┴─────────────────────┘
```

### 4.2 配置启用

```conf
aof-use-rdb-preamble yes
```

### 4.3 优势

- **恢复速度**：先快速加载 RDB 前缀，再重放少量 AOF 命令
- **数据安全**：AOF 增量部分保证最近写入不丢失
- **体积优化**：RDB 前缀天然紧凑，比纯 AOF 更小

## 📊 五、RDB vs AOF 全面对比

### 5.1 性能对比

| 指标 | RDB | AOF |
|:---|:---|:---|
| 写入性能 | 无持续开销，fork 时短暂阻塞 | 持续追加，everysec 几乎无感知 |
| 恢复性能 | **O(1) 加载**，与数据量无关 | O(N) 重放，命令越多越慢 |
| 磁盘 IO | 集中式爆发写入 | 持续追加写入 |
| 内存开销 | fork 时可能翻倍（COW） | 缓冲区 + 重写时 fork |

### 5.2 数据安全对比

| 场景 | RDB | AOF (everysec) | AOF (always) |
|:---|:---|:---|:---|
| 正常关机 | 丢最后一次快照后数据 | 丢最多 1 秒 | 零丢失 |
| 异常断电 | 同上 | 可能丢 1-2 秒 | 最多丢 1 条 |
| 磁盘满 | 快照失败，数据完整 | 追加失败，可能丢数据 | 同上 |

### 5.3 选型决策树

```
是否需要最高数据安全性？
├── 是（金融、订单）
│   └── AOF (always) + RDB 备份
│       └── 注意：吞吐量下降 50%+
│
└── 否
    ├── 内存数据可重建？（纯缓存）
    │   ├── 是 → 仅 RDB 或 不持久化
    │   └── 否
    │       ├── 数据量 < 10GB，写入适中？
    │       │   ├── 是 → RDB + AOF (everysec) 【推荐】
    │       │   └── 否 → 分片 + AOF + 定时 RDB 冷备
    │       └── 超大内存（> 50GB）？
    │           └── Redis Cluster 分片，每节点 < 10GB
    │
    └── 恢复速度优先？（灾难恢复）
        └── RDB + AOF 混合模式（Redis 7.0+）
```

## ⚙️ 六、生产环境推荐配置

### 6.1 通用配置（同时开启 RDB + AOF）

```conf
# ========== RDB 配置 ==========
save 900 1
save 300 10
save 60 10000

# ========== AOF 配置 ==========
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes

# AOF 重写
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 256mb

# ========== 内存优化 ==========
maxmemory 8gb
maxmemory-policy allkeys-lru

# 关闭 THP，避免 fork 延迟
# 操作系统层面：echo never > /sys/kernel/mm/transparent_hugepage/enabled
```

### 6.2 关键监控指标

| 指标 | 获取方式 | 告警阈值 |
|:---|:---|:---|
| `rdb_last_bgsave_time_sec` | INFO persistence | > 1 秒 |
| `aof_current_size` / `aof_base_size` | INFO persistence | 膨胀率 > 200% |
| `aof_buffer_length` | INFO persistence | > 1MB |
| `aof_delayed_fsync` | INFO stats | > 0 |

## 🚀 七、实战案例：秒杀系统的 Redis 持久化策略

### 7.1 业务背景

秒杀系统具有以下典型特征：
- **瞬时超高并发**：秒杀开始时 QPS 可达百万级
- **库存准确性敏感**：超卖 = 业务灾难
- **热点 Key 集中**：大量用户同时访问同一商品库存
- **时间窗口短**：秒杀通常持续几秒到几分钟

### 7.2 架构设计：缓存层与持久化层分离

```
┌─────────────────────────────────────────────────────────┐
│                      用户请求层                          │
│         Nginx → Gateway → 限流/防刷 → 秒杀服务           │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│  ★ Redis 缓存层（高性能，弱持久化，可重建）               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐  │
│  │  库存预扣减  │  │  用户去重   │  │   限流计数器     │  │
│  │  stock:1001 │  │ bought:uid  │  │   limit:1001    │  │
│  │   (String)  │  │   (Set)     │  │    (String)     │  │
│  └─────────────┘  └─────────────┘  └─────────────────┘  │
│  持久化策略：RDB 定时 + AOF everysec（混合模式）           │
│  目标：快速恢复，允许秒级数据丢失                          │
└─────────────────────────────────────────────────────────┘
                          │
                          ▼ (异步)
┌─────────────────────────────────────────────────────────┐
│  ★ MySQL 订单层（强一致性，最终持久化）                   │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐  │
│  │  真实库存表  │  │  订单表      │  │   秒杀记录表     │  │
│  │   (InnoDB)  │  │  (InnoDB)   │  │    (InnoDB)     │  │
│  └─────────────┘  └─────────────┘  └─────────────────┘  │
│  持久化：Binlog + 主从复制（强一致）                      │
│  目标：最终一致性，零数据丢失                             │
└─────────────────────────────────────────────────────────┘
```

**核心设计思想**：Redis 负责高性能的"资格筛选"，MySQL 负责最终的"强一致持久化"。

### 7.3 持久化策略的特殊考量

#### 7.3.1 秒杀期间禁用自动 RDB

在高并发秒杀期间，自动 RDB 的 `BGSAVE` 会触发 `fork()` 系统调用：

- `fork()` 本身需要复制页表，耗时与内存大小成正比
- Copy-On-Write 机制下，高写入频率会导致内存急剧膨胀
- 主线程阻塞期间，大量秒杀请求会超时或失败

**策略**：关闭自动 `save` 条件，改为秒杀结束后手动触发 `BGSAVE`。

#### 7.3.2 AOF 刷盘策略选择

| 策略 | 秒杀场景适用性 | 原因 |
|:---|:---|:---|
| `always` | ❌ 禁用 | 每条命令 fsync，吞吐量暴跌，无法支撑高并发 |
| `everysec` | ✅ 推荐 | 每秒后台刷盘，最多丢 1 秒数据，性能与安全平衡 |
| `no` | ❌ 不推荐 | 可能丢失大量数据，秒杀结果不可接受 |

#### 7.3.3 混合模式的优势

启用 `aof-use-rdb-preamble yes` 后：
- 恢复时先快速加载 RDB 前缀部分
- 再重放少量的 AOF 增量命令
- 兼顾了恢复速度和数据安全性

### 7.4 三层防护架构

#### 第一层：Redis 原子扣减

使用 Lua 脚本在 Redis 端原子执行以下操作：
1. 检查用户是否已购买（幂等去重）
2. 原子扣减库存（防止超卖）
3. 记录用户购买状态
4. 设置过期时间（用于未支付释放）

Lua 脚本在 Redis 中是原子执行的，天然避免竞态条件。同时，AOF 会记录整个 Lua 脚本而非展开后的每条命令，有效减少 AOF 文件体积。

#### 第二层：MQ 异步落库

扣减成功的请求通过消息队列异步发送到订单服务：
- 数据库层面使用事务保证订单创建的原子性
- 数据库的 Binlog + 主从复制提供强一致性持久化
- 即使 Redis 在秒杀期间发生故障，MySQL 中的真实库存和订单记录不会丢失

#### 第三层：数据预热与灾难恢复

**预热阶段**（秒杀开始前）：
- 从 MySQL 加载真实库存到 Redis
- 此时系统负载低，可以安全执行 `BGSAVE` 生成基准 RDB
- 设置合理的 Key 过期时间（如秒杀结束后自动清理）

**灾难恢复**（Redis 故障重启后）：
- Redis 通过 RDB/AOF 恢复后，数据可能不是最新的（AOF everysec 最多丢 1 秒）
- 以 MySQL 中的真实库存和订单记录为准，重建 Redis 缓存
- 重建内容包括：剩余库存、已购用户集合等

### 7.5 完整时间线策略

```
秒杀开始前 30 分钟：
    │
    ▼
┌─────────────────┐
│  预热 Redis 库存 │ ← 从 MySQL 加载真实库存
│  执行 BGSAVE    │ ← 生成基准 RDB（低峰期安全操作）
└─────────────────┘
    │
    ▼
秒杀进行中（高流量）：
    │
    ├── Redis 配置：
    │   ├── AOF: appendfsync everysec（每秒刷盘）
    │   ├── RDB: 禁用自动 save（避免 fork 阻塞）
    │   └── 混合模式: aof-use-rdb-preamble yes
    │
    ├── 业务层：
    │   ├── Lua 原子脚本扣减库存
    │   ├── 异步 MQ 发送订单消息
    │   └── 限流/熔断保护 Redis
    │
    └── 监控告警：
        ├── Redis 内存使用率 < 80%
        ├── 慢查询 > 1ms 告警
        └── AOF 重写进度监控
    │
    ▼
秒杀结束后：
    │
    ├── 手动触发 BGSAVE（生成最终快照）
    ├── AOF 重写（压缩日志体积）
    └── 数据归档到冷存储
```

### 7.6 关键设计原则总结

| 原则 | 说明 |
|:---|:---|
| **Redis 是筛选器，不是仓库** | 真实订单和库存以 MySQL 为准 |
| **允许秒级数据丢失** | 通过 AOF everysec 平衡性能，丢失数据可从 MySQL 重建 |
| **秒杀期间禁用 RDB** | 避免 fork 阻塞导致请求超时 |
| **Lua 保证原子性** | 避免并发扣减导致超卖 |
| **异步落库削峰** | MQ 缓冲高并发写入，保护数据库 |

## ❓ 八、常见问题 FAQ

### Q1：RDB 和 AOF 可以同时开启吗？

**可以，且推荐同时开启。** Redis 4.0+ 支持同时启用，恢复时优先使用 AOF（数据更完整）。混合模式（Redis 7.0+）更是将两者优势结合。

### Q2：AOF 重写会阻塞主线程吗？

**不会。** AOF 重写通过 `fork()` 子进程完成，与 `BGSAVE` 类似。但重写期间的新写入会同时写入旧 AOF 和 AOF 重写缓冲区，重写完成后合并。

### Q3：大内存实例（> 10GB）如何优化持久化？

- 考虑 Redis Cluster 分片，每个节点控制在 4-8GB
- 禁用自动 RDB，改用手动 + AOF everysec
- 关闭 THP（透明大页），减少 fork 延迟
- 使用 SSD 磁盘，提升 AOF 刷盘性能

### Q4：为什么 AOF 恢复比 RDB 慢？

RDB 是直接加载二进制到内存，时间复杂度 O(1)。AOF 需要逐条解析并执行命令，时间复杂度 O(N)，且命令越多越慢。混合模式通过 RDB 前缀大幅缓解这个问题。
